# 转换规则

> **阶段1 脚本设计文档**——阶段2 AI 正常处理无需阅读此文件。仅当脚本行为异常需要排查时查阅。

脚本阶段（阶段1）的核心转换规则，包括前页识别、TOC 处理和标题编号还原。

## 1. 前页区检测

### 1.1 分界判定

从文档首行开始逐行向后扫描，遇到以下信号时停止扫描，其后内容整体视为正文保留：

- 遇到 ATX 标题行（`# ...`），且该标题不属于 TOC 锚点标记
- 遇到连续 2 行及以上不含封面关键词、TOC 模式、旧式表格分隔符的普通文本段落

### 1.2 前页噪音类型

| 类型 | 特征 |
|------|------|
| 封面 | 加粗项目名/模块名、`(内部文档,不得复制)`、公司名 |
| 审批表 | 旧式管线表格（拟制/审核/批准 + 人员 + 日期） |
| 修订记录表 | 旧式管线表格（版本/作者/日期/说明） |
| TOC 块 | 锚点标记 + 嵌套链接条目（含页码） |

### 1.3 前页移除规则

1. 仅在文档起始连续区域内识别前页候选块，不跨过已确认的正文段落
2. 优先删除可明确判定的块：封面关键词块、审批/修订表、TOC 锚点与 TOC 条目块
3. 若起始区域出现无法判定的普通段落或非 TOC 标题，则停止向后删除，后续内容整体保留
4. 正文首个 Heading 仅作为辅助信号，不作为"其前全部删除"的唯一依据

## 2. TOC 块检测

从前页区尾部向前扫描，匹配以下模式：

| 名称 | 正则 | 示例 |
|------|------|------|
| TOC 条目 | `\[[\d.]+ .+? \[\d+\]\(#[^)]*\)\]\(#[^)]*\)` | `[1 介绍 [5](#介绍)](#介绍)` |
| TOC 锚点 | `\[\]\{#_Toc\d+ \.anchor\}` | `[]{#_Toc28717 .anchor}` |

## 3. 标题编号提取

在删除 TOC 之前，从 TOC 条目中提取有序映射。映射键包含"标题层级 + 标题文本 + 出现序号"，以避免重复标题冲突：

```
不带尾点的编号风格：

TOC 条目 "[1 介绍 [5](#介绍)](#介绍)"
  → { level: 1, title: "介绍", ordinal: 1, number: "1" }

TOC 条目 "[1.1 总体说明 [5](#总体说明)](#总体说明)"
  → { level: 2, title: "总体说明", ordinal: 1, number: "1.1" }

TOC 条目 "[2.1 总体说明 [9](#总体说明)](#总体说明)"
  → { level: 2, title: "总体说明", ordinal: 2, number: "2.1" }

带尾点的编号风格：

TOC 条目 "[1. 介绍 [5](#介绍)](#介绍)"
  → { level: 1, title: "介绍", ordinal: 1, number: "1." }

TOC 条目 "[1.1. 目的 [5](#目的)](#目的)"
  → { level: 2, title: "目的", ordinal: 1, number: "1.1." }

TOC 条目 "[4.6.1.3.1. 模块1安全设计 [20](#模块1安全设计)](#模块1安全设计)"
  → { level: 5, title: "模块1安全设计", ordinal: 1, number: "4.6.1.3.1." }
```

**level 计算**：按 `.` 分隔数字段计数（忽略尾点），如 `1.1.` 含两个数字段 → level 2，`4.6.1.3.1.` 含五个数字段 → level 5。

**正文侧 ordinal 匹配**：扫描正文标题行时，按 `(level, normalized_title)` 维护出现计数器。每遇到一个标题，先将该组合的计数器 +1 得到当前 ordinal，再到 TOC 有序映射中查找匹配的条目消耗其编号。

## 4. 标题编号还原

### 4.1 标题层级下移

由于输出 Markdown 的 H1 用于文档标题（文件名去扩展名），Pandoc 输出的所有内容标题需整体下移一级：

```
Pandoc H1 (#)    →  输出 H2 (##)
Pandoc H2 (##)   →  输出 H3 (###)
Pandoc H3 (###)  →  输出 H4 (####)
Pandoc H4 (####) →  输出 H5 (#####)
Pandoc H5 (#####)→  输出 H6 (######)
```

支持最多 5 级编号深度（对应 H2-H6）。若原文档层级超过 5 级（Pandoc 输出 H6），下移后将超出 Markdown 标准的 6 级 `#` 限制，此时保持 H6 不再下移，并在 JSON 报告的 `attention` 字段中记录。

### 4.2 编号还原策略（按优先级）

1. **TOC 有序映射法**（首选）：利用第 3 节提取的有序条目，按"标题层级 + 规范化标题文本 + 出现序号"逐个匹配标题行并补回编号
2. **层级计数法**（TOC 不可用时回退）：按标题层级维护各级计数器，逐个标题行生成编号
   - 遇到某级标题时，该级计数器 +1，同时将所有更低级（子级）计数器归零
   - 遇到跳级（如 H1 直接到 H3，跳过 H2）时，缺失的中间层级计数器视为 1，继续向下编号
   - 若标题文本已以数字编号开头（如 `## 2.1 功能需求`），识别为已编号标题，直接保留不重复添加
3. **不还原**：若两种方法均不可靠（如 TOC 缺失且标题中混有已编号和未编号），保留 Pandoc 原始输出，交由 AI 后处理阶段判断

**编号格式保留**：若 TOC 条目中编号带尾点（如 `1.`、`1.1.`），还原时保留尾点；否则不添加。不同文档的编号风格可能不同，应原样保留。

### 4.3 还原结果验证

还原完成后，检查编号序列的连续性：

- 相邻同级标题编号应递增且无跳号（如 `2.1` 后应为 `2.2` 而非 `2.4`）
- 子级编号应在父级编号前缀下（如 `3.1.x` 应出现在 `## 3.1` 之后）
- 若发现断号或层级不匹配，在 JSON 报告的 `attention` 字段中记录异常位置，不自动修正
