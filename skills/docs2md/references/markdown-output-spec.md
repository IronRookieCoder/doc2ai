# Markdown 输出规范

> **按需查阅**——阶段2 靶向修复协议已内含操作指引，正常处理无需预先阅读此文件。仅当需确认完整输出规范或验收标准时查阅。

## 定位与目标

本文档定义 docs2md 最终 `.md` 文件的质量标准，是阶段2（AI 后处理）的执行依据和验收基准。

- **读者**：执行阶段2 的 AI、审核转换结果的人
- **上游输入**：阶段1 脚本产出的 `.md`
- **下游产出**：最终 `.md` 文件（可选附带 `.json` 报告）
- **核心原则**：信息抽取与结构重组，非版式还原。宁可多保留不可误删

## 全局约束（必须遵守）

以下约束在任何处理步骤中均不可违反，按优先级从高到低排列：

1. **不修改实质内容**——仅做格式优化，不改写术语、数字、日期、约束条件
2. **不新增内容**——不添加原文不存在的结论、归纳或章节标题
3. **不确定时保留并标记**——无法判断是否应删除/修改时，保留原文并在报告 `attention` 中记录
4. **不修改文档级标题**——H1 由脚本从文件名生成，AI 不额外新增或修改
5. **不主动调整标题层级**——仅在存在明确结构错误（如跳级）时修正，不因美观主动整体下沉/上提

## 内容取舍标准

### 保留项

| 内容类型 | 说明 |
|---------|------|
| 文档级标题（H1） | 来自文件名去扩展名，阶段1 已生成 |
| 标题层级与章节编号 | 包括阶段1 还原的编号 |
| 正文段落 | 所有非前页区的正文内容 |
| 模板指导语 | 斜体标记（`*...*`），描述该章节应填写的内容，包含 AI 理解文档边界的关键信息 |
| 表格 | GFM pipe 表格；复杂表格由阶段1 脚本压平为规则矩形 |
| 约束/注释 | 加粗约束、脚注等 |
| URL | 保留为裸 URL 文本，不转为 Markdown 链接语法 |

### 删除项

| 内容类型 | 说明 |
|---------|------|
| 封面/审批页/修订记录 | 阶段1 已删除，AI 仅处理残留 |
| 目录（TOC） | 阶段1 已删除，AI 仅处理残留 |
| 页眉页脚 | Pandoc 天然忽略，无需处理 |
| 图片及引用 | 阶段1 已删除，AI 仅处理残留（如 `![alt](path)` 遗漏） |
| 水印/页码 | 若残留则删除 |
| 空段落 | 连续空行压缩为单个空行 |
| 格式噪音 | Pandoc 标注（`{.mark}`、`{.underline}`）、空锚点等 |

## 阶段1 脚本处理范围

以下操作由阶段1 脚本执行，阶段2 AI 只处理异常残留。

### 表格格式标准化

Pandoc 转换阶段通过 `scripts/gfm-table-normalize.lua` 将 Pandoc `Table` AST 统一改写为 GFM pipe table。

**转换规则**：
- simple table、old-style table、grid table 均输出为 GFM pipe table
- GFM pipe table 无法表达合并单元格，跨行/跨列内容压平为规则矩形：左上角保留内容，其余覆盖位置补空
- 多段落、列表、代码块等块级内容折叠为单格文本，块之间用 `<br>` 或 `<br><br>` 连接
- 无表头表格补充空表头，满足 GFM pipe table 语法要求
- caption 转为表格前的普通文本行 `Table: ...`
- 表格内 `<br>` 保留；表格内 `&nbsp;` 保留实体形式，不做 unescape

## AI 后处理范围

以下操作需要语义判断，脚本无法可靠处理，由阶段2 AI 执行。

### 1. 断裂段落合并

软回车（Shift+Enter）导致的段落拆分，将属于同一语义段落的多行合并为一段。

**判断依据**（同时满足才合并）：
- 前一行不以句号 `。`、分号 `；`、冒号 `：` 等终结标点结尾
- 后一行不以标题标记 `#`、列表标记 `- / * / 1.`、表格分隔符 `|` 开头
- 两行在语义上构成连续表述（非独立子句）

**正例**：
```
# 合并前
规划表、用户需求、系统需求、需求矩阵几个文档之间，
所述的同一个需求点，名称要一致，以方便需求跟踪和评审。

# 合并后（.md）
规划表、用户需求、系统需求、需求矩阵几个文档之间，所述的同一个需求点，名称要一致，以方便需求跟踪和评审。
```

**反例（不应合并）**：
```
# 这两行是独立的列表项，不应合并
*1、需求关联性，内部功能依赖，外部跨团队协作*
*2、对公司其他产品或组件的影响*
```

### 2. 表格异常残留处理

阶段1 已负责表格格式标准化。若阶段2 仍发现 grid table（`+---+` 格式）、old-style table（`--- ---` 格式）、HTML `<table>` 或 pipe table 列对齐错误，视为脚本异常残留或复杂表格降级风险。

**判断依据**：
- 能确认是简单格式残留且不改变实质内容时，可修正为 GFM pipe table
- 涉及复杂合并关系、内容位置不确定或无法机械判断时，保留原文并在 `attention` 中记录
- 表格内 `<br>` 保留（合法 GFM 格式，用于单元格内换行）
- 表格内 `&nbsp;` 保留实体形式，不做 unescape

### 3. 标题层级修正

仅在发现**明确跳级错误**时修正（如 H2 直接到 H4，中间缺少 H3）。

**禁止**：因美观或统一性主动整体调整标题层级。

### 4. 残留噪音识别

脚本未覆盖的边缘噪音模式，识别后移除：
- 非标准格式的封面残留
- Pandoc 标注遗漏（`{.mark}`、`{.underline}` 等）
- 图片占位空表格

### 5. 模板指导语斜体还原

Pandoc 可能丢失部分段落的斜体标记。当同一章节内其他模板指导语均为斜体（`*...*`），且某段落在语义上明确属于模板指导语（描述"该章节应填写什么"），应补充斜体标记。

**判断依据**：上下文中同类段落均有斜体标记，且该段落不是正文内容。

### 6. 可疑内容记录

发现可疑问题时记入报告 `attention` 字段，**不默认修改正文**：
- 标题编号序列异常（跳号、重复）
- 标题层级超出 H6 限制
- 疑似残留的非正文内容

## 输出文件命名

输出文件沿用原始 `.doc` / `.docx` 文件名，仅替换扩展名：

| 阶段 | 文件名 |
|------|--------|
| 最终输出 | `原文件名.md` |

示例：`XX项目_系统需求规格说明书_V6.0.1.doc` → `XX项目_系统需求规格说明书_V6.0.1.md`

## 结果报告格式（可选）

当脚本使用 `--report` 参数或 AI 阶段需要记录变更时，生成 JSON 报告：

```json
{
  "file_name": "文件名.doc",
  "conversion_result": "success | failed",
  "script_result": {
    "status": "success | failed",
    "output_md": "路径/文件名.md",
    "preconverted_docx": "路径/文件名.docx"
  },
  "ai_result": {
    "status": "success | failed",
    "output_md": "路径/文件名.md",
    "changes": [
      {"type": "修改类型", "section": "涉及章节", "detail": "具体描述"}
    ]
  },
  "attention": ["需要人工关注的问题"]
}
```

**字段规则**：
- `attention`：仅在存在内容疑点时填充，无疑点时为空数组或省略
- `script_result.preconverted_docx`：仅在输入为 `.doc` 且预转换成功时出现
- `ai_result.changes`：记录 AI 阶段所有格式修改，每条包含类型、章节、详情
- `conversion_result`：`success` 表示两阶段均成功；任一阶段失败则为 `failed`

## 验收标准

最终 `.md` 文件通过以下检查即为合格：

| # | 检查项 | 判定方式 |
|---|--------|---------|
| 1 | H1 标题为文件名去扩展名 | 首行匹配 `# 原文件名` |
| 2 | 正文标题从 H2 开始 | 无 H1 级正文标题 |
| 3 | 标题编号正确还原 | 编号与 TOC/文档结构一致 |
| 4 | 无封面/审批/TOC 残留 | 不含审批表、修订记录、TOC 锚点 |
| 5 | 无图片引用残留 | 不含 `![` 模式 |
| 6 | 无 Pandoc 噪音 | 不含 `{.mark}`、`{.underline}`、空锚点 |
| 7 | 表格为 GFM pipe 格式 | 不含 `+---+` grid table、`--- ---` old-style table 或 HTML `<table>` |
| 8 | 模板指导语保留斜体 | 斜体标记 `*...*` 完整 |
| 9 | 无断裂段落 | 同一语义段落在同一行内 |
| 10 | 实质内容未被修改 | 与原文逐段比对，仅格式变化 |
